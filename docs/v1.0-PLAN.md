### Sprint 3: Server-Side Mode (Basic)

**Duration**: 2-3 weeks  
**Branch**: `feature/server-mode`

#### Feature Specification

**Core Requirements**
- Page-based pagination
- Cursor-based pagination
- Infinite scroll with fetchMore
- Debounced sort/filter/search to backend
- Loading states (skeleton rows, spinners)
- Error handling with retry
- Server-side grouping flag

**API Design**
```typescript
// Props
interface TableProps {
  serverMode?: boolean;
  serverData?: ServerDataAdapter;
  paginationMode?: 'page' | 'cursor' | 'infinite';
  debounceMs?: number;
}

// Types
interface ServerDataAdapter {
  fetch: (params: ServerQueryParams) => Promise<ServerResponse>;
  fetchMore?: (cursor: string) => Promise<ServerResponse>;
}

interface ServerQueryParams {
  page?: number;
  pageSize?: number;
  cursor?: string;
  sort?: SortParam[];
  filter?: FilterParam[];
  search?: string;
}

interface ServerResponse {
  rows: T[];
  totalCount?: number;
  hasNextPage?: boolean;
  cursor?: string;
  error?: string;
}

// Events
interface TableEvents {
  'server-error': (error: Error) => void;
  'server-loading': (loading: boolean) => void;
}
```

## Risk Mitigation

### High-Risk Technical Areas

Based on ARCHITECTURE.md warnings:

1. **Dynamic Row Height + Virtualization**
   - Risk: Layout thrashing, incorrect scroll positions
   - Mitigation: Create torture test early, consider fixed-height mode only for v1.0

2. **Sticky Columns + Horizontal Virtualization**
   - Risk: Position calculation complexity
   - Mitigation: Defer horizontal virtualization to v1.1, use CSS sticky only

3. **Resizing + Pinning**
   - Risk: Width calculation conflicts
   - Mitigation: Comprehensive test suite, clear constraints

### Risk Register

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Memory leaks in production | High | Medium | Phase 0 focus, stress testing |
| Performance regression | High | Low | Benchmark harness, CI monitoring |
| Accessibility non-compliance | High | Medium | Early testing, screen reader audit |
| Feature scope creep | Medium | High | Strict v1.0 scope, defer to v1.1+ |
| Documentation delays | Medium | Medium | Parallel documentation sprints |
| WASM complexity | Medium | Low | Defer to v1.1, JS-first approach |

### Branch Strategy
```
main (stable)
  └── develop (integration)
       ├── feature/row-expansion
       ├── feature/grouping-aggregation
       ├── feature/server-mode
       ├── feature/accessibility
       └── feature/documentation
```

### Commit Convention
```
feat(scope): description
fix(scope): description
docs(scope): description
test(scope): description
refactor(scope): description
chore(scope): description
```

### Pull Request Process
1. Create feature branch from develop
2. Implement feature with tests
3. Update documentation
4. Submit PR with description
5. Code review required
6. All tests must pass
7. Coverage must not decrease
8. Squash merge to develop
